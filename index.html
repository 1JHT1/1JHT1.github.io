1.快速排序

这种基于递归的方法在数据量级大的时候容易出现栈溢出。原因是：每次递归调用都会在调用栈上压入一个新的栈帧（包含局部变量、返回地址等），而栈内存大小有限（通常几MB）。如果递归深度过大（如快速排序最坏情况递归深度 O(n)），栈帧数量超过栈容量，就会抛出 StackOverflowError。迭代版本使用堆内存模拟栈，可以避免此限制。

使用显式栈模拟递归过程，将每次分区的左右边界入栈，循环处理，从而避免递归调用。这就是迭代优化方案。

关于浏览器外部链接跳转失效的问题解决

问题：在浏览器外的所有连接，在点击后不能直接跳转到浏览器进行浏览，包括支付链接，内容链接等。

起初的最暴力的解决是直接关闭后台进程：

进到任务管理器，找到浏览器的后台进程进行强制关闭，但这会导致浏览器的固定页面也会全部丢失，相当于重新打开浏览器，里面固定的所有快捷页面都会丢失，这十分影响我需要同时固定若干页面以便快速获取内容信息。

后来，当我使用git bash推送内容的时候出现了一个推送失败的问题：

这个问题表示 Git 无法与远程仓库建立网络连接，出现问题的可能原因如下：

  • DNS 解析错误（比如 hosts 文件将 github.com 指向 127.0.0.1）
  • 代理配置错误或代理服务器未运行
  • 防火墙或网络策略阻止了对 443 端口的出站连接
  • GitHub 服务器不可达（极少见）

由于我使用了Steam++(Watt Toolkit)的“加速”功能，默认使用 hosts 劫持 的方式，把大量域名的解析结果强制改为 127.0.0.1，然后它自己运行一个本地代理服务来转发请求。然而，它可能并没有覆盖所有被劫持的域名，或者代理服务没有正确处理某些跳转链接，导致这些域名实际上无法被访问。

Steam++的hosts文件放在C:\Windows\System32\drivers\etc\hosts下，在这里可以获得一个如下的劫持文件

浏览器的 DNS 缓存：重启浏览器或系统后才会生效。

为什么 Steam++ 会导致外部链接无法跳转？

问题现象
  • 点击某些外部链接（例如跳转到 github.com）时，浏览器无反应或提示无法连接。
  • 删除 Steam++ 添加到 hosts 文件中的规则后，跳转恢复正常。
根本原因

Steam++ 的 hosts 劫持生效，但它的本地代理服务未正常运行。

Steam++ 的加速原理

  1. 修改 hosts 文件
    将大量域名（如 github.com、steamcommunity.com 等）强制解析到本地回环地址 127.0.0.1。

  2. 启动本地代理服务
    在后台监听 127.0.0.1 的 80、443 等端口，捕获指向本机的请求，并通过“加速通道”转发到真实服务器。

当时失败的状态
  • hosts 劫持已生效
    系统将目标域名解析为 127.0.0.1。
  • 本地代理服务未运行
    可能原因：未点击“加速”按钮、服务崩溃、端口被占用、防火墙拦截等。
  • 结果
    浏览器尝试连接 127.0.0.1:443，但没有任何程序监听该端口 → 连接被拒绝（ERR_CONNECTION_REFUSED）→ 页面无法打开，跳转无响应。
为什么删除 hosts 规则后恢复？
  • 移除 127.0.0.1 映射后，系统恢复正常 DNS 解析，直接访问真实服务器 IP，不再依赖本地代理。
  • 连接成功，跳转恢复正常。
如何复现该问题（测试用）
  1. 在 hosts 文件中添加测试行：
    127.0.0.1 www.baidu.com
  2. 确保没有代理服务监听 80/443 端口：
    • 退出 Steam++、Clash 等所有代理软件。
    • 关闭浏览器的 DNS over HTTPS（DoH）。
    • 关闭系统代理。
  3. 清除 DNS 缓存：ipconfig /flushdns。
  4. 在浏览器中访问 http://www.baidu.com → 应显示“无法连接”。
  5. 点击外部链接（指向 www.baidu.com）→ 跳转失败。
为什么你现在重新添加 hosts 规则却仍能跳转？

可能存在以下绕过 hosts 的机制：

  • 浏览器开启了 DNS over HTTPS (DoH)，加密 DNS 查询不读取系统 hosts。
  • 启用了 TUN 模式（虚拟网卡），网络流量被其他代理接管。
  • 系统代理或 VPN 仍在生效，覆盖了 hosts 设置。
总结
  • 失败条件 = hosts 劫持生效 + 本地无代理监听对应端口。
  • 恢复正常 = 移除 hosts 劫持 或 启动本地代理服务 或 通过其他方式绕过 hosts。
  • 若当前环境能正常跳转，说明存在某种机制接管了网络请求，使得 hosts 劫持被忽略。

本文档解释了外部链接无法跳转的根本原因及测试方法。

重启后不能链接跳转的核心原因

之前：hosts 劫持生效 + Steam++ 代理服务未运行 → 连接失败
现在：hosts 劫持生效 + Steam++ 代理服务自动运行 → 连接成功

你看到的 hosts 文件中，Steam++ 已经把 github.com 等域名都指向了 127.0.0.1（你自己的电脑）。这意味着任何访问这些域名的请求，都会被发往本机的某个端口（比如 33210）。

Steam++ 加速的原理是：它同时启动一个本地代理服务，监听 127.0.0.1:33210，专门接收这些被劫持的请求，然后通过它的“加速通道”转发到真实的服务器。

  • 以前重启后不能跳转：因为 Steam++ 没有开机自启，所以重启后 hosts 劫持虽然在，但代理服务没运行。浏览器访问 github.com 时，系统解析到 127.0.0.1:33210，发现那里没有程序在监听，导致连接被拒绝（ERR_CONNECTION_REFUSED），所以页面打不开，外部链接也无法跳转。
  • 现在开机自启后能跳转：系统一启动，Steam++ 就自动运行并开启代理服务，监听在 127.0.0.1:33210。当浏览器再次访问被劫持的域名时，请求成功被本地代理接收并转发出去，因此能正常跳转。

1.hasNext 和 hasNextLine 的区别？

方法 判断依据 返回 true 的条件
hasNext() 默认以空白字符（空格、制表符、换行符等）作为分隔符 输入中还存在下一个 token（例如，一个单词、一个数字）
hasNextLine() 以换行符（\n 或 \r\n）作为分隔符 输入中还存在下一行（无论该行是否为空）

也就是说数据文件里要是按照空格分开数组用hasNext(),如果用行来区别数据就用hasNextLine()判断是否存在

2.next和nextInt方法区别？

方法 返回类型 读取内容 停止条件
next() String 读取下一个 token（以空白分隔的字符串） 遇到空白符（空格、换行等）结束
nextInt() int 读取下一个整数值 遇到非数字字符结束（但要求下一个 token 必须是整数，否则抛出 InputMismatchException）

3.按行输入nextLine()

按行输入对应的方法是 nextLine()，它会读取当前行的剩余内容（包括空格，直到换行符），并返回该行字符串（不包含换行符）。

  • next() 和 nextInt() 不会消耗行尾的换行符，因此如果在它们之后调用 nextLine()，可能会直接读取到一个空行。

解决方法：在 nextInt() 之后加一个 sc.nextLine() 来消耗掉换行符。

4.不同的文件读取方式不同

方式 示例方法 读取单位 适用场景
按字节读取 InputStream.read() 字节 二进制文件、原始数据
按字符读取 Reader.readLine() 字符 文本文件
按 token 读取 Scanner.nextLine() 整数 token（字符序列） 结构化文本输入

给定一个 *m* x *n* 的矩阵，如果一个元素为 0 ，则将其所在行和列的所有元素都设为 0 。请使用 原地 算法**。**

一个直观的解决方案是使用 O(*m**n*) 的额外空间，但这并不是一个好的解决方案。

一个简单的改进方案是使用 O(*m* + *n*) 的额外空间，但这仍然不是最好的解决方案。

你能想出一个仅使用常量空间的解决方案吗?

这里使用了两个bollean数组，巧妙的避开了在原数组上在循环过程中修改值导致其他值变化的连锁反应，也就是说，如果我们不希望在循环的过程中修改值导致一些列的连锁反应的时候，我们可以去构造额外的空间去记载或者过渡这些需要修改的值，然后再次遍历的时候进行修改！空间换时间 → 用 O(m+n) 的额外空间换取清晰的逻辑和正确的置零顺序，避免了 O(m×n) 的复制或复杂的原地标记。

时间复杂度：O(nm)、空间复杂度O(m+n)；

54—螺旋矩阵

给你一个 m 行 n 列的矩阵 matrix ，请按照 顺时针螺旋顺序 ，返回矩阵中的所有元素。

思路解析：

我们在矩阵数组中进行螺旋遍历的时候，有两个问题需要解决，一个是要知道自己什么时候要转向，另一个是怎么指定一套四个位置的转向方法。

对于已经访问过的数字，可将其标记为 ∞ 或者空，从而避免重复访问。用一个长为 4 的方向数组 DIRS=[(0,1),(1,0),(0,−1),(−1,0)] 分别表示右下左上 4 个方向。同时用一个下标 di 表示当前方向，初始值为 0，表示一开始向右。每次移动，相当于把行号增加 DIRS[di][0]，把列号增加 DIRS[di][1]。
向右转 90 ∘相当于把 di 增加 1，但在 di=3 时要回到 di=0。两种情况合二为一，把 di 更新为 (di+1)mod4。
