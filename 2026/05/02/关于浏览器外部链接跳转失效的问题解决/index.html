关于浏览器外部链接跳转失效的问题解决

问题：在浏览器外的所有连接，在点击后不能直接跳转到浏览器进行浏览，包括支付链接，内容链接等。

起初的最暴力的解决是直接关闭后台进程：

进到任务管理器，找到浏览器的后台进程进行强制关闭，但这会导致浏览器的固定页面也会全部丢失，相当于重新打开浏览器，里面固定的所有快捷页面都会丢失，这十分影响我需要同时固定若干页面以便快速获取内容信息。

后来，当我使用git bash推送内容的时候出现了一个推送失败的问题：

这个问题表示 Git 无法与远程仓库建立网络连接，出现问题的可能原因如下：

  • DNS 解析错误（比如 hosts 文件将 github.com 指向 127.0.0.1）
  • 代理配置错误或代理服务器未运行
  • 防火墙或网络策略阻止了对 443 端口的出站连接
  • GitHub 服务器不可达（极少见）

由于我使用了Steam++(Watt Toolkit)的“加速”功能，默认使用 hosts 劫持 的方式，把大量域名的解析结果强制改为 127.0.0.1，然后它自己运行一个本地代理服务来转发请求。然而，它可能并没有覆盖所有被劫持的域名，或者代理服务没有正确处理某些跳转链接，导致这些域名实际上无法被访问。

浏览器的 DNS 缓存：重启浏览器或系统后才会生效。

为什么 Steam++ 会导致外部链接无法跳转？

问题现象
  • 点击某些外部链接（例如跳转到 github.com）时，浏览器无反应或提示无法连接。
  • 删除 Steam++ 添加到 hosts 文件中的规则后，跳转恢复正常。
根本原因

Steam++ 的 hosts 劫持生效，但它的本地代理服务未正常运行。

Steam++ 的加速原理

  1. 修改 hosts 文件
    将大量域名（如 github.com、steamcommunity.com 等）强制解析到本地回环地址 127.0.0.1。

  2. 启动本地代理服务
    在后台监听 127.0.0.1 的 80、443 等端口，捕获指向本机的请求，并通过“加速通道”转发到真实服务器。

当时失败的状态
  • hosts 劫持已生效
    系统将目标域名解析为 127.0.0.1。
  • 本地代理服务未运行
    可能原因：未点击“加速”按钮、服务崩溃、端口被占用、防火墙拦截等。
  • 结果
    浏览器尝试连接 127.0.0.1:443，但没有任何程序监听该端口 → 连接被拒绝（ERR_CONNECTION_REFUSED）→ 页面无法打开，跳转无响应。
为什么删除 hosts 规则后恢复？
  • 移除 127.0.0.1 映射后，系统恢复正常 DNS 解析，直接访问真实服务器 IP，不再依赖本地代理。
  • 连接成功，跳转恢复正常。
如何复现该问题（测试用）
  1. 在 hosts 文件中添加测试行：
    127.0.0.1 www.baidu.com
  2. 确保没有代理服务监听 80/443 端口：
    • 退出 Steam++、Clash 等所有代理软件。
    • 关闭浏览器的 DNS over HTTPS（DoH）。
    • 关闭系统代理。
  3. 清除 DNS 缓存：ipconfig /flushdns。
  4. 在浏览器中访问 http://www.baidu.com → 应显示“无法连接”。
  5. 点击外部链接（指向 www.baidu.com）→ 跳转失败。
为什么你现在重新添加 hosts 规则却仍能跳转？

可能存在以下绕过 hosts 的机制：

  • 浏览器开启了 DNS over HTTPS (DoH)，加密 DNS 查询不读取系统 hosts。
  • 启用了 TUN 模式（虚拟网卡），网络流量被其他代理接管。
  • 系统代理或 VPN 仍在生效，覆盖了 hosts 设置。
总结
  • 失败条件 = hosts 劫持生效 + 本地无代理监听对应端口。
  • 恢复正常 = 移除 hosts 劫持 或 启动本地代理服务 或 通过其他方式绕过 hosts。
  • 若当前环境能正常跳转，说明存在某种机制接管了网络请求，使得 hosts 劫持被忽略。
