2.3 Lua 脚本中的冲突

Lua 脚本通过 redis.call('GET', stockKey) 获取到的是上述 JSON 字符串，而不是纯数字。tonumber() 解析 JSON 字符串会返回 nil，导致 stock 变量为 nil。

虽然脚本中有 if stock == nil then return -2，但由于 Redis 中缓存了旧版本的脚本（缺少该判断），或者实际脚本行数与预期不符，错误发生在后续的 if stock < quantity 比较中，因为 nil 无法与数字比较，从而抛出 attempt to compare number with nil。

3. 解决方案

核心思路：与 Lua 脚本交互的 Redis 键（如库存 key、用户订单 key）必须使用 纯字符串 存储，不能使用 Jackson 序列化。

StringRedisTemplate 默认使用 StringRedisSerializer，所有值都会以字符串形式存储，完美适配 Lua 脚本。

6. 总结

  • 问题本质：Spring Data Redis 的 Jackson 序列化机制导致存储的整数被转换为 JSON 字符串，与 Lua 脚本期望的纯数字字符串不兼容。
  • 解决关键：区分业务场景，对需要与 Lua 脚本交互的 Redis 键使用 StringRedisTemplate 确保纯字符串存储。
  • 最佳实践：在使用 Lua 脚本时，始终使用 StringRedisTemplate 操作相关键，避免序列化干扰。

通过以上调整，秒杀功能可正常执行原子扣库存、防重下单，并异步生成订单。
