138—随机链表的复制

给你一个长度为 n 的链表，每个节点包含一个额外增加的随机指针 random ，该指针可以指向链表中的任何节点或空节点。构造这个链表的 深拷贝。 深拷贝应该正好由 n 个 全新 节点组成，其中每个新节点的值都设为其对应的原节点的值。新节点的 next 指针和 random 指针也都应指向复制链表中的新节点，并使原链表和复制链表中的这些指针能够表示相同的链表状态。复制链表中的指针都不应指向原链表中的节点 。例如，如果原链表中有 X 和 Y 两个节点，其中 X.random --> Y 。那么在复制链表中对应的两个节点 x 和 y ，同样有 x.random --> y 。返回复制链表的头节点。用一个由 n 个节点组成的链表来表示输入/输出中的链表。每个节点用一个 [val, random_index] 表示：val：一个表示 Node.val 的整数。random_index：随机指针指向的节点索引（范围从 0 到 n-1）；如果不指向任何节点，则为 null 。你的代码 只 接受原链表的头节点 head 作为传入参数。

如果是普通的链表，直接按照遍历顺序创建链表即可，但由于这里的节点涉及到随机指针，在我们拷贝节点的时候，当前节点的随机节点指针可能还未创建，故利用回溯的思路，让每个节点的拷贝操作相互独立，对于当前节点，我们首先拷贝，然后进行后继节点和随机指针指向节点的拷贝，拷贝完成就将创建的新节点指针返回，故完成了当前节点的两个指针的赋值。

用HashMap记录每一个节点对应的新节点的创建情况，遍历链表过程中，检查当前节点的后继和随机指针创建的节点清空，如果其中一个没有创建，则立刻递归地创建！这里很巧妙的去递归的实现未创建节点然后进行拷贝，当我们拷贝完成，会回溯到当前层继续进行拷贝完成当前节点的拷贝。

特别注意的是一个节点可能被多个其他节点指向，因此我们在递归的时候为了防止重复拷贝，先检查一遍当前节点是否被拷贝过，已经拷贝则直接在HashMap中返回拷贝后的节点指针。

回溯+HashMap，时间复杂度O(n)、空间复杂度O(n)：

下面用另一种想法来省去用HashMap的空间：

在我们复制前将原来链表的每一个节点都赋值一个在其后，例如A->B->C变成A->A’->B->B’->C->C’，这种情况下我们可以你直接获得拷贝节点的下一个随机节点理应的对应节点（由于保持了原来链表的特性），故每一个拷贝节点的随机下一个节点在原链表已经锁定，但是随机下一个节点可能为null

当完成随机的后，我们只要将这个链表按原节点和拷贝节点拆分，遍历一次，需要注意最后一个拷贝节点的后继节点为null

迭代+节点拆分：时间复杂度O(n)、空间复杂度O(1)

42—接雨水

给定 n 个非负整数表示每个宽度为 1 的柱子的高度图，计算按此排列的柱子，下雨之后能接多少雨水。

分析：我们把每一个节点看成是一个两侧有板的桶，这个桶的装水量是当前的短板去掉桶里的石头，这两个板子的最大值在移动的时候需要进行维护！因为水流只会往低处流，那么这个桶的左右板可以看出是它局部的左右最大高度！那么在这样的结构下我们在左右两边设置双指针进行遍历，当left<=right的时候，如果左边的板高度大于右边的板，那么用右边的板减去当前的高度（石头）即为这个桶的装水量，随后right指针往左移一格表示当前桶已计算。反之，用左边的板高度减去当前的高度。left指针向右移动。一直遍历下去并累加会得到整个模型能装的水总量。

名词介绍

2026-04-17

并发量：每一秒的请求数(QPS)

TPS：每一秒的事务数

ACK：确认。用于确认收到数据。在三次握手中，服务器收到 SYN 后回复 SYN+ACK 表示“我收到了你的请求，这是我的序列号，并且确认你的序列号”。最后客户端再发 ACK 确认，连接建立。

SYN： 同步序列号。用于发起一个新连接。在三次握手中，客户端发送 SYN 包给服务器，表示“我想建立连接，这是我的初始序列号”。

FIN: TCP 协议中用于 终止连接 的控制报文，全称为 Finish（结束）。当一个通信方完成数据发送后，会发送 FIN 包，表示“我没有数据要发了，请求关闭连接”。

transient: 是一个关键字，用于修饰类的成员变量（字段）。它的作用是：当该类的对象被序列化时，被 transient 修饰的字段不会参与序列化过程。换句话说，这个字段的值在序列化时会被忽略，反序列化后会被重置为默认值（如 null、0、false 等）

序列化 是指将程序中的对象（内存中的数据结构）转换为字节序列（或文本格式如 JSON/XML）的过程，以便通过网络传输或保存到文件中。反序列化则是逆向过程，将字节流恢复为原来的对象。

protected 成员可以被同一个包中的任何类以及不同包中的子类访问。

平衡二叉搜索树（Balanced Binary Search Tree）是一种特殊的二叉搜索树，它同时满足以下两个核心特性：
1.叉搜索树特性：对于树中的每个节点，其左子树的所有节点值都小于该节点值，右子树的所有节点值都大于该节点值（左小右大）。
2.平衡性：树中任意节点的左右两个子树的高度差的绝对值不超过 1。这保证了树的结构不会出现过度倾斜的情况。

1.TCP三次握手？四次挥手

三次握手

首先TCP是面向连接的协议，使用前必须先在客户端和服务端建立连接，而这个连接正是三次握手的用途。初始时刻，服务端和客户端处于Close状态，然后服务端主动监听某个端口，处于Listen状态；

客户端随机初始化一个【序列号】client_isn，同时把SYN标志位置为1，发动第一个SYN报文，表示向服务端发起连接，之后客户端处于SYN_Sent状态；

服务端收到客户端报文后也会随机初始化自己的【序列号】server_isn,并且将【应答号】设置为客户端的序列号+1，同时修改SYN和ACK标志位置为1；随后服务端会发送第二个报文 SYN+ACK报文给客户端，状态变成SYN_Received状态；

客户端收到服务端报文后，将【应答号】修改为服务端【序列号】+1，并将ACK标志为1；然后客户端向服务端回复第三个ACK报文，之后修改为Established状态；

服务端收到客户端的ACK报文后同步进入Established状态。

Q1为什么不是二次，要三次？（无法防止历史连接，会导致服务器错误建立连接造成资源浪费）
四次挥手

初始时刻，客户端和服务端都处于Established状态，客户端打算关闭连接，那么将FIN标志位置为1，发送第一个FIN报文，表示关闭连接，随后状态处于等待关闭状态；

【同一个类内部调用（没有走代理）】。
同一个类中的无事务方法直接调用本类的 @Transactional 方法，不走代理，事务失效。
例如：orderService.createOrder() 内部调用 this.updateStock()（带事务注解），
更新库存不会开启新事务。

【方法非 public（代理无法拦截）。】
@Transactional 只能作用于 public 方法，否则代理无法拦截。

【异常被 catch 后没有重新抛出。】
代码中 try-catch 了异常但没有 throw，框架感知不到异常，不会回滚。

【数据库引擎不支持事务（MyISAM）。】
MySQL 表使用了 MyISAM 引擎（常见于旧项目或建表语句遗漏），该引擎不支持事务。需改为 InnoDB。

【配置了错误的传播行为（如 SUPPORTS）。】
SUPPORTS 不会主动创建事务，在无事务环境下执行等于“事务失效”了。

####@Autowired是Spring框架，@Resource是JDK标准下的

当一个接口有多个实现类时，只用 @Autowired 会报错，必须配合 @Qualifier；而 @Resource 默认会按字段名去匹配，如果字段名与某个实现类 Bean 名称一致，会自动注入。

@Resource 如果没指定 name，则先按字段名匹配，失败再按类型匹配；而 @Autowired 始终先按类型。
####@Autowired 是 Spring 按类型注入的首选；@Resource 是 JDK 标准，按名称注入更直接。混合使用时注意各自的行为差异。

【不同之处】
（1）默认注入方式不同，@autowired是根据类型注入（bytype）@resource是根据名称（byname）
（2）指定名称@autowired需要结合@qualifier(“name”) 而resource直接指定@resource(name="")
（3）@Autowired支持require属性：
@Resource 没有 required 属性，如果指定了 name 但 Bean 不存在，也会抛异常，无法像 @Autowired 那样允许为 null。
